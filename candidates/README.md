# Candidates

Tracking which Lean packages we are considering for inclusion in
`lean-pool`, along with the reasoning behind each decision.

## Layout

- `manual.txt` (committed) — hand-edited list of GitHub URLs for
  packages that Reservoir does not index (forks, missing
  `lake-manifest.json`, non-OSI license, fewer than two stars). One
  URL per line; `#` comments and blank lines allowed.
- `CRITERIA.txt` (committed) — the inclusion policy used when
  classifying a candidate as in / out of `lean-pool`.
- `decisions.jsonl` (committed) — one classifier verdict per line
  (`repo`, `include`, `category`, `confidence`, `rationale`). Source
  of truth for what shows up in the table below.
- `raw_data/` (gitignored) — local-only data fetched by
  `python -m lean_pool.aggregator fetch`: the Reservoir manifest plus
  GitHub metadata for each entry in `manual.txt`.
- The summary table below is regenerated from those files by
  `python -m lean_pool.aggregator render`. By default it shows only
  packages that pass three filters: ≥ 2 stars (Reservoir's own floor),
  ≥ 250 lines of `.lean` code in the local clone, and `include=true`
  in `decisions.jsonl` (repos with no verdict are kept regardless).
  The LOC column counts every `.lean` file under the clone outside
  `.lake`/`.git`/`build` directories. The In Pool column shows ✓ for
  candidates already imported into lean-pool — it's joined from the
  `source.github_repo` field of each project in `LeanPool/projects.yml`.
  Pass `--min-stars=0 --min-loc=0 --decisions /dev/null` to disable all
  three filters and render the full list.

Do not edit the autogenerated section by hand.
